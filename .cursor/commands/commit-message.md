# commit-message

## 目的

ステージ済み差分からコミットメッセージ案を作成し commit する。

- 変更内容を要約する。
- コミットメッセージ案を複数提示する。
- ユーザー承認があるまで `git commit` は実行しない。
- ユーザー指示があるまで `git push` は絶対に実行しない。

## このリポジトリの前提

このリポジトリは ccfoliaCSSPreviewer（ココフォリア向けカスタム CSS プレビュー用 Electron アプリ）である。

技術スタック: Electron Forge + Vite + React + TypeScript + Tailwind + jotai + shadcn

主な構成:

- `src/main.ts`: Electron メインプロセス
- `src/preload.ts`: preload スクリプト
- `src/renderer.tsx` / `src/App.tsx`: React レンダラー
- `src/components/`: UI・プレビュー関連コンポーネント
- `src/atoms/`: jotai 状態
- `src/data/`: CSS リスト等のデータ
- `src/lib/`: ユーティリティ
- `forge.config.ts` / `vite.*.config.*`: Electron Forge + Vite 設定

標準コマンド:

- `npm start`: Electron アプリ起動（Forge）
- `npm run lint`: ESLint（`.ts` / `.tsx`）
- `npm run package` / `npm run make`: パッケージング / 配布物作成

注意:

- 専用の typecheck / test スクリプトは `package.json` に無い。
- 検証コマンドを実行した場合のみ「実行済み」と書く。
- 実行していない検証を、実行済みのように書かない。
- UI 確認は原則 `npm start` での目視確認とする。

## 最初に確認すること

まず、必ず以下を確認する。

```bash
git status --short
git branch --show-current
git log -1 --format="%an %ae"
git config user.name
git config user.email
git log -n 20 --pretty=format:'%s'
```

## コミット作者チェック

このPCは複数ユーザーで使われる可能性があるため、コミットメッセージ案を出す前に、必ずコミット作者を確認する。

確認する値:

- 直近コミット作者: `git log -1 --format="%an %ae"`
- 現在の Git ユーザー名: `git config user.name`
- 現在の Git メールアドレス: `git config user.email`

判断ルール:

1. `git config user.name` または `git config user.email` が空の場合は、コミットメッセージ案より前に「Gitユーザー設定が不足している」と明記する。
2. 直近コミット作者と現在の Git 設定が異なる場合は、「作者設定が直近コミットと異なる」と明記する。
3. 作者設定が異なっていても、コミットメッセージ案は作成してよい。
4. ただし、作者設定が要確認の状態では、ユーザーが明示的に承認するまで `git commit` は実行しない。
5. 作者設定の修正は、ユーザーから明確な指示がある場合だけ行う。
6. 名前からメールアドレスを推測しない。
7. メールアドレスを変更する場合は、ユーザーがメールアドレスを明示している必要がある。
8. 名前とメールアドレスの両方を変更する場合は、両方の値が明示されている場合だけ実行する。
9. グローバル設定ではなく、このリポジトリだけの設定として扱う。`--global` はユーザーが明示した場合だけ使う。

作者チェックの報告例:

```text
# 作者チェック:
- 直近コミット作者: 紅坂柚月 akasaka@example.com
- 現在のGit設定: 紅坂柚月 akasaka@example.com
- 判定: OK。直近コミット作者と現在設定が一致しています
```

作者が違う場合:

```text
# 要確認:
直近コミット作者と現在のGit設定が異なります。
このままコミットすると、別ユーザー名で記録される可能性があります。

- 直近コミット作者: 紅坂柚月 akasaka@example.com
- 現在のGit設定: 別ユーザー other@example.com
```

## 作者設定を直す指示の扱い

作者設定を直す場合は、変更したい項目に合わせて解釈する。

### 名前だけ直す場合

ユーザー指示例:

```text
git config user.name を紅坂柚月に直して
```

実行するコマンド:

```bash
git config user.name "紅坂柚月"
```

この場合、`user.email` は変更しない。

### メールアドレスだけ直す場合

ユーザー指示例:

```text
git config user.email を akasaka@example.com に直して
```

実行するコマンド:

```bash
git config user.email "akasaka@example.com"
```

この場合、`user.name` は変更しない。

メールアドレスが明示されていない場合は実行しない。

### 名前とメールアドレスの両方を直す場合

ユーザー指示例:

```text
git config user.name を紅坂柚月、user.email を akasaka@example.com に直して
```

実行するコマンド:

```bash
git config user.name "紅坂柚月"
git config user.email "akasaka@example.com"
```

名前とメールアドレスの両方が明示されている場合だけ実行する。

### 曖昧な指定の場合

次のような指定は、名前とメールアドレスのどちらをどう変更するか判断できないため、そのまま実行しない。

```text
git config user.name と user.email を紅坂柚月に直して
```

理由:

- `user.name` に設定する値は名前である。
- `user.email` に設定する値はメールアドレスである。
- `紅坂柚月` はメールアドレスではない。
- メールアドレスは名前から推測してはいけない。

この場合は、次のように聞き返す。

```text
# 確認:
名前だけを紅坂柚月に直しますか？
メールアドレスも直す場合は、設定するメールアドレスも指定してください。

## 例:
- 名前だけ: git config user.name を紅坂柚月に直して
- メールだけ: git config user.email を akasaka@example.com に直して
- 両方: git config user.name を紅坂柚月、user.email を akasaka@example.com に直して
```

### 単発コミットだけ作者を指定する場合

リポジトリ設定を変更せず、今回のコミットだけ作者を指定する場合は `--author` を使う。

```bash
git commit --author="名前 <メールアドレス>" -m "[type] コミットメッセージ"
```

ただし、これもユーザーが名前とメールアドレスの両方を明示している場合だけ提案・実行する。

## 差分確認

次に、ステージ済み差分と未ステージ差分を確認する。

```bash
git diff --cached --stat
git diff --cached --name-status
git diff --stat
git diff --name-status
```

必要に応じて、内容把握のために以下も確認する。

```bash
git diff --cached
git diff
```

判断ルール:

1. ステージ済みの変更がある場合は、原則としてステージ済み差分をコミット対象として扱う。
2. ステージ済みの変更がなく、未ステージ差分だけがある場合は、コミットメッセージ案を作成しない。
3. ステージ済みの変更がなく、未追跡ファイルだけがある場合も、コミットメッセージ案を作成しない。
4. ステージに上げ忘れている可能性がある場合は、作者設定の要確認と同じ扱いで、その時点で処理を終了する。
5. 未ステージ差分や未追跡ファイルを勝手に `git add` しない。
6. 複数の責務・機能・領域が混在しており、分けてコミットした方がよい場合は、コミットメッセージ案を作成しない。
7. 分けてコミットした方がよい場合は、分割案を提示して、その時点で処理を終了する。
8. 変更がない場合は、コミットメッセージを作らず「変更がありません」と報告する。

## コミットメッセージの基本形式

このリポジトリでは、近年の実コミット履歴に合わせて、1行目は以下の形式を使う。

```text
[type] 日本語の要約
```

Issue と関係がある場合の1行目は、以下の形式にする。

```text
[type] #Issue番号 日本語の要約
```

例:

```text
[fix] #9 Webview のミュート処理を修正
```

コミットメッセージは、原則として複数行で作成する。

```text
1行目: 変更内容の要約（タイトル、概要）
2行目: 空行
3行目以降: 変更した理由、内容、詳細
```

例:

```text
[fix] Webviewをミュート

プレビュー再生時に音声が鳴らないよう、Webview をミュートする。
```

```text
[add] jotaiで状態管理

カスタム CSS やタブ選択などの状態を jotai で管理し、ローカルストレージへ永続化できるようにする。
```

1行目の参考例:

```text
[add] カスタムCSSを追加
[add] jotaiで状態管理
[add] jotai
[fix] Webviewをミュート
[fix] メニュー削除
[fix] タイトルを変更
[fix] shadcn/ui のcursorを調整
[fix] パスの移動
[update] レイアウト変更
[remove] 未使用コンポーネントを削除
```

## type の選び方

| type | 用途 | 例 |
|------|------|----|
| `[add]` | 新機能、新ページ、新API、新コンポーネント、新しい仕組みの追加 | `[add] jotaiで状態管理` |
| `[fix]` | 不具合修正、型安全性改善、責務整理、命名修正、配置修正、既存実装の問題解消 | `[fix] Webviewをミュート` |
| `[update]` | 既存機能・見た目・データ・設定の更新 | `[update] レイアウト変更` |
| `[remove]` | 未使用コード、不要ファイル、不要設定の削除 | `[remove] 未使用コンポーネントを削除` |
| `[docs]` | ドキュメントのみの変更 | `[docs] 開発コマンドの説明を更新` |
| `[chore]` | 依存関係、環境設定、ビルド設定など、機能に直結しない保守 | `[chore] npm 依存関係を更新` |

優先順位:

1. 新しく使えるものが増えたなら `[add]`
2. 壊れていたもの、誤っていたもの、責務や型の問題を直したなら `[fix]`
3. 既存仕様を調整しただけなら `[update]`
4. 消しただけなら `[remove]`
5. ドキュメントだけなら `[docs]`
6. 判断が割れる場合は、最新の `git log -n 20 --pretty=format:'%s'` の傾向に寄せる。

## 表記ルール

- コミットメッセージは日本語を基本にする。
- 末尾に句点「。」は付けない。
- 1行目は30〜45文字程度を目安に圧縮する。
- 2行目は必ず空行にする。
- 3行目以降には、変更した理由・内容・詳細を簡潔に書く。
- `説明` はユーザー確認用の説明であり、推奨コミットメッセージ本文とは別に出力する。
- 推奨コミットメッセージの引用欄には、実際に `git commit` へ渡す複数行メッセージだけを出す。
- 英語の技術名・固有名詞は無理に日本語化しない。
- 既存履歴に合わせ、英語識別子の前後には必要に応じて半角スペースを入れる。
- Issue と関係がある変更の場合は、1行目の `[type]` 直後に `#123` のように Issue 番号を入れる。
- Issue 番号付きの形式は `[fix] #9 修正内容` のようにする。
- Issue 番号は、ブランチ名・会話・ユーザー指示・差分から明確に分かる場合だけ入れる。
- Issue 番号が不明な場合は、推測して入れない。
- Issue 番号はコミット本文の末尾には置かない。
- Qiita記事の例にある `#{ticket_no}` 形式は、このリポジトリの主流ではないため、明確な指示がない限り使わない。

固有名詞の表記例:

- Electron
- Vite
- React
- TypeScript
- Tailwind
- jotai
- shadcn
- Webview
- CcfoliaNav
- CssWebview
- UI
- API
- HTML
- CSS

## 要約時に見る観点

変更内容を確認するときは、以下を優先して見る。

- どの領域か: `src/main.ts`, `src/preload.ts`, `src/components/`, `src/atoms/`, `src/data/`, `src/lib/`, Forge / Vite 設定
- 何が変わったか: 追加、修正、更新、削除、移動、リネーム
- なぜ必要そうか: 不具合解消、責務整理、UI改善、型安全性、プレビュー精度、状態管理
- 影響範囲: メインプロセス、レンダラー、Webview、ナビ、タブ、CSS リスト、jotai atom など
- 検証状況: 実行したコマンド、未実行の理由、実行不能だった理由
- 作者設定: 直近コミット作者と現在の Git 設定が一致しているか

## 出力形式

### 出力の見た目ルール

- 出力見出しは Markdown 見出し形式にする。
- 主要見出しは `# 見出し:` の形式にする。
- 補助見出しが必要な場合は `## 見出し:` の形式にする。
- 引用形式 `> ` を使うのは、原則として `# 推奨コミットメッセージ:` の中だけにする。
- `# 推奨コミットメッセージ:` では、実際に `git commit` へ渡す複数行メッセージ全体を引用形式で出力する。
- コミットメッセージ内の空行も `>` だけの行で表現する。
- `# 推奨コミットメッセージ:` 以外の `# コミットの概要:`、`# 説明:`、`# 理由:`、`# 確認した内容:` などは引用にしない。
- 箇条書きは通常の `-` を使う。

コマンド実行時は、必ず以下の形式で出力する。

```text
# 推奨コミットメッセージ:
> [fix] XXXXX
>
> Webview のミュート処理を修正し、プレビュー再生時に音声が鳴らないようにします。

# コミットの概要:
Webview のミュート処理を修正し、プレビュー再生時に音声が鳴らないようにします。

# 説明:
- `src/components/CssWebview.tsx` の Webview 設定を調整します
- 既存実装の不整合を直す変更が中心です
- プレビュー再生時の音声挙動に影響します
- この説明はユーザー確認用であり、通常の Git commit body には含めません

# 理由:
- 変更の中心が既存実装の修正なので `[fix]` が最も自然です
- 影響範囲は `src/components/CssWebview.tsx` です
- 直近のコミット履歴でも `[fix] 日本語要約` の形式が使われています

# 他の候補:
1. [fix] XXXXX

   Webview のミュート設定の不整合を修正します。

2. [update] XXXXX

   Webview の再生時設定を調整します。

3. [fix] XXXXX

   プレビュー再生時の音声ミュート処理を整理します。

# 確認した内容:
- ブランチ: main
- 対象差分: ステージ済み差分
- 主な変更ファイル: src/components/CssWebview.tsx
- 検証: 未実行。今回はコミットメッセージ作成のみのため

# 作者チェック:
- 直近コミット作者: XXXXX xxxxx@example.com
- 現在のGit設定: XXXXX xxxxx@example.com
- 判定: OK。直近コミット作者と現在設定が一致しています

# コミット実行の確認:
推奨案でコミットする場合は「OK。推奨案でコミット」と指示してください。
別案でコミットする場合は「OK。2でコミット」のように指示してください。
```

Issue と関係がある場合は、推奨コミットメッセージの1行目で `[type]` の直後に Issue 番号を入れる。

```text
# 推奨コミットメッセージ:
> [fix] #9 Webview のミュート処理を修正
>
> プレビュー再生時に音声が鳴らないよう、Webview をミュートする処理を整理します。
```

ステージ済みの変更がない場合は、コミットメッセージ案を出さず、その時点で終了する。

```text
# 要対応:
ステージ済みの変更がありません。
未ステージ差分または未追跡ファイルがあるため、コミット対象を確定できません。

# 確認した内容:
- ブランチ: main
- ステージ済み差分: なし
- 未ステージ差分: あり
- 未追跡ファイル: あり

# 次の対応:
コミット対象に含めるファイルをステージしてから、再度このコマンドを実行してください。

例:
git add path/to/file
```

分けてコミットした方がよい場合は、コミットメッセージ案を出さず、その時点で終了する。

```text
# 要対応:
複数の責務が混在しているため、分けてコミットした方が安全です。

# 分割案:
1. Webview ミュート処理の修正
   - 対象: src/components/CssWebview.tsx
   - type 候補: [fix]

2. ドキュメント更新
   - 対象: .cursor/commands/...
   - type 候補: [docs]

# 次の対応:
上記の単位に分けてステージしてから、再度このコマンドを実行してください。
```

作者設定に問題がある場合は、`# 推奨コミットメッセージ:` より前に出す。

```text
# 要確認:
直近コミット作者と現在のGit設定が異なります。
このままコミットすると、別ユーザー名で記録される可能性があります。

# 確認した内容:
- 直近コミット作者: 紅坂柚月 akasaka@example.com
- 現在のGit設定: 別ユーザー other@example.com

# 次の対応:
先に作者設定を直す場合は、変更したい項目に合わせて指定してください。

- 名前だけ: git config user.name を紅坂柚月に直して
- メールアドレスだけ: git config user.email を akasaka@example.com に直して
- 名前とメールアドレスの両方: git config user.name を紅坂柚月、user.email を akasaka@example.com に直して
```

## コミット実行ルール

ユーザーが明示的に承認するまで、`git commit` を実行してはならない。

承認例:

- `OK。推奨案でコミット`
- `OK。1でコミット`
- `OK。2でコミット`
- `そのメッセージでコミットして`

承認後の動作:

1. `git status --short` で直前の状態を再確認する。
2. `git log -1 --format="%an %ae"` と `git config user.name` / `git config user.email` で作者設定を再確認する。
3. 作者設定が要確認のまま、ユーザーが作者不一致を明示的に了承していない場合は、`git commit` を実行せずに確認する。
4. ステージ済み差分がない場合は、未ステージ差分や未追跡ファイルがあっても `git commit` を実行しない。
5. ステージ済み差分がない場合は、勝手に `git add .` や `git add -A` を実行しない。
6. 分けてコミットした方がよい状態の場合は、分割案を提示して処理を終了する。
7. ステージ済み差分があり、分割不要と判断できる場合だけ、承認された複数行メッセージで `git commit` する。
8. 複数行コミットは `git commit -m "1行目" -m "3行目以降"` の形式で実行する。
9. Issue 番号を含める場合は、1行目の `[type]` 直後に `#123` のように入れる。例: `[fix] #9 修正内容`
10. コミット後に `git log -1 --pretty=format:'%h %an %ae %B'` で結果を確認して報告する。
11. `git push` は絶対に実行しない。

コミット後の報告例:

```text
# コミット完了:
abc1234 [fix] XXXXX

# 作者:
紅坂柚月 akasaka@example.com

# push:
実行していません。
```

## エラー対応

`git commit` が失敗した場合は、以下を報告する。

- 実行したコマンド
- エラーメッセージ
- 現在の `git status --short`
- 現在の `git config user.name`
- 現在の `git config user.email`
- 次に取るべき対応案

代表例:

- ステージ済みファイルがない
- pre-commit hook で失敗した
- コンフリクト中である
- Git user.name / user.email が未設定
- 改行コードや権限変更だけが大量に出ている

## NG

- ユーザー承認前に `git commit` しない。
- ユーザー指示なしに `git push` しない。
- ユーザー指示なしに `git add .` しない。
- ステージ済み差分がない状態で、コミットメッセージ案を作成しない。
- ステージ済み差分がない状態で、`git commit` を実行しない。
- 分けてコミットした方がよい状態で、コミットメッセージ案を作成しない。
- 分けてコミットした方がよい状態で、`git commit` を実行しない。
- ユーザー指示なしに Git の作者設定を変更しない。
- メールアドレスを名前から推測しない。
- 名前とメールアドレスをまとめて指定された場合に、勝手に両方へ同じ値を設定しない。
- 変更内容を見ずにコミットメッセージを作らない。
- `git log -1 --format="%an %ae"` と `git config user.name` / `git config user.email` を確認せずにコミットしない。
- 実行していないテストや型チェックを「実行済み」と書かない。
- このリポジトリの実履歴に合わない `feat:` / `fix:` / `chore:` 形式を優先しない。
- 明確な根拠なく Issue 番号を付けない。
- Issue 番号をコミット本文の末尾に置かない。
- `# 推奨コミットメッセージ:` 以外の出力本文を引用形式にしない。
